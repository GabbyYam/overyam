Classes and Interfaces

最小化类及其成员的访问权限（accessibility）

  • 隐藏内部实现，只提供外部接口（information hiding）

    • 易测试
    • 易重用
    • 提升组件的独立性

  • 基本原则：尽可能的让类及其属性无法被外部访问

  • 对于Top-level 的类和接口：有两种可能

    • package-private：那么，可以在保证不影响外部使用的情况下
    • public：必须永远保证维护时不影响其他与该类相关类的使用

  • 如果一个Top-level类只有一个类继承它，考虑将其写成静态内部类放入该类中

  • 对于成员和接口，则有如下四种可能：

    • private：无外部访问权限
    • package-private：包内
    • public：all classes
    • protected：子类和包内

  • public型的API完成之后，请将不需要外部调用的部分全部改为private

  • 实例字段最好不要public，因为容易导致线程不安全（not thread safe），而应该为static final（当然这样一来就牺牲了灵活性了

  • 如果是列表（Array）的话，不要加static final，因为这个字段很可能是要改动的

==Extra：==module system in Java 9 - 待补全

public类中，用getter和setter访问private字段

  • 对于需要访问类内字段的类，加入getter来进行数据访问

  • 对于需要可变的类字段，提供setter来修改内部值

尽可能的不提供外部修改权限

如果内部的字段不需要修改，那就不要提供setter

实现immutable class的几个原则：

  • 不要提供修改内部状态的method

  • 保证该类无法被继承（extends）

  • 所有字段均设为private final

  • 优点：如下

    • 线程安全
    • 简单
    • 可以自由的共享数据

优先采用composition而不是继承

继承的天然缺陷：

  • 子类会依赖于父类，所以一旦父类改动，子类功能就会收到影响
  • 如果父类新增了一个恰巧和子类同名的method，那么子类直接过不了编译
  • 有可能会暴露不必要的API给子类，比如Java中的Properties继承了Hashtable导致Properties可以调到put和get方法，就很离谱了

什么情况下不适于继承：

  • 如果某个类并不是设计成可用来继承的类，并且method没有做好相关的说明，那么最好不要乱继承

复合如何实现：forwarding class & forwarding methods

本质上是装饰者模式，详细参考设计模式-装饰者模式

要么为继承而设计并提供说明文档，要么就禁止继承

那么要设计成易继承的类呢？应该好好写好document标注相关注意事项

如何document？：

  • 必须标注可重写的方法中调用了哪些其他可重写方法，以及后续产生什么效果，以防止继承后调用错乱
  • 写清楚实现细节

一些原则：

  • Constructor中不能调overridable method
  • 如果实现cloneable & serializable接口，clone & readObject方法均不可调overridable method

如何禁止继承：

  • final
  • private & package-private

优先写接口而不是抽象类

  • 接口的优点：

    • 可以多实现，更加灵活
    • 容易更新
    • 允许实现非层次型的结构

  • 非要用的话可以考虑接口结合抽象类

接口设计

由于接口可以写default method，所以注意以下几点：

  • default mathod 会影响许多方法，可能会导致已有的实现类失效，或添入不必要的麻烦
  • default method 可以不必让实现类重写，但是需要注意上一条

常量：

  • 如果是强相关的，写成类的Enum成员
  • 否则将常量定义在接口里

优先选择静态内部类

nested class 有以下几种：

  • 静态内部类：工具类、辅助类
  • 内部类
  • 匿名内部类：动态构建 functional object、lambda表达式
  • local 类（本地类）：论外，应该用不到吧

静态内部类与内部类区别在于：

  • 非静态的内部类相关于具体对象，而静态内部类只相关于类本身
  • 非静态类的instance需要依赖于外部类的instance
    • 非静态内部类适合做什么？：Adapter，参见适配器模式，用来使外部类被看作一个不相关的类
    • 为什么尽量不要non-static的原因：对于每个外部类instance，内部类都会产生一个相关的instance，这样会导致额外的空间和时间消耗

重写 equals

不需要重写equals的情况：

  • 对于每一个对象实例应该是unique的时
  • 根据API设计的需要，没必要提供逻辑上的相等时
  • 父类已经重写equals方法且子类直接继承就合适时
  • 外部没有权限访问该类的内部方法时

重写需要满足一下性质

  • 自反性：self equals self时必须返回true
  • 对称性：如果A equals B返回true 那么反之也要返回true
  • 传递性：A equals B 且B equals C，那么A也必须equals C
  • 不变性：每次调用应该返回相同的结果
  • 非空性：对于所有非空的对象equals null 必须返回false

如果不满足以上性质会怎么样？有空的时候思考一下，待补全。

高质量的equals实现代码的一些tips：

  • 用 == 来判定参数是否是一个对象的引用，如果是则返回true（自反性），此为一种效率上的优化，因为做全面的对比代价可能较大

  • 用 instanceof 去判断参数的类型，如果类型不对那么直接返回false

  • 如果通过的instanceof的测试，将参数强转为对应类的对象以保证下面的检测正确的进行

  • 对于所有有意义的字段field进行比对，如果matches则通过返回true，否则返回false

Create&Destroying Objects

静态工厂StaticFactory

  • 注意这里说的静态工厂不是设计模式里的工厂方法
  • 优点：主要归为以下几点
    • 名字好看，而不是和构造器一样是个类名加括号
    • 不需要每次都返回一个新的实例，而是有已经创建好的缓存实例（类似单例的思想？）
    • 可以返回任何返回类型的子类型，使得程序灵活性更好
    • 基于上者，静态工厂在互相调用的过程中，返回类型可以不断改变
    • 编写改静态工厂时，返回的对象所属的类可以不必存在（也就是可以先写静态工厂再写类）
  • 缺点：主要依赖于构造器Constructor
    • 如果该类没有public或protected构造器的话，就无法返回子类了
    • 一般不会写在API文档里，所以会比较难找

建造者Builder

当构造器参数非常多的时候，考虑使用Builder

  • Java Bean的弊端：
    • Bean在构造过程中可能会产生数据不一致
    • Bean不能将类做成immutable，这样就必须得考虑线程安全的问题了
  • Builder的优点
    • 可读性强，写起来也很方便
    • 非常适合类继承：抽象类有抽象类的Builder，实现类有实现类的Builder，实现类的Builder可以直接继承父类的Builder，再重写build方法即可，灵活易于扩展

单例设计

用private Constructor或Enum type来实现单例属性

  • 优点：

    • 每次返回相同的对象引用
    • 更加简洁
    • 在不改变API接口的情况下，更灵活的在单例和非单例之间切换（静态工厂static factory）
    • 为了保证反序列化后仍然返回相同的instance，添加readResolve method或Enum type的

    注意当该类要 extends 别的类时，不能使用Enum

适当使用依赖注入 dependency injection

如果某类可能依赖多项基础资源underlying resource，不要采用单例或是static的方式去依赖资源，而是采用依赖注入的方式，这样可能通过更改配置文件的方式，更灵活且testable得来对外部资源进行依赖

不要创建没用&多余的objects

  • 能引用常量时不要采用new的方式
  • 能用基本类型时不要使用包装类
  • string.match隐含了每次执行都要new一个pattern，因此要注意避免此类情况产生
  • 当然有时根据需要可能要new新的object而不是复用已有的object

删除不需要的reference

目的是防止内存泄漏，让内存被gc回收

内存泄漏的常见来源：

  • 缓存cache：比如HashMap缓存时，其实可以用WeakHashMap来替代会更好，这样可以被自动gc
  • 监听和回调(listener and callbacks)：简单的解决办法是全部采用弱引用（weak reference）的方式

尽量不使用finalizer & cleaner

cleaner是Java9引入的finalizer的替代品，两者相当于C++中的destructor（析构函数）

不使用的原因&缺点：

  • 因为他们都具有不可预测性，因此如果在不恰当的时间回收了资源，就会发生错误

  • 如果finalizer的优先级很低，可能资源在很长一段时间都不会被回收，最后outOfMemoryError

相关分析比较复杂，待补充

永远要用try-with-resource而不是try-finally来链接资源

所谓try-with-resource即将资源的连接写在try(…)括号内，而不是写在finally域内

  • 一来如果要开启多个resource，那么写在finally内很丑很长很麻烦
  • 写在try括号内，可读性更强
  • 前者的写法同样可以达到相同的效果

面向接口编程

如何保证复用代码的同时，让代码有效的分配到你希望的实现类中呢？

答案是面向接口编程，如果你想要分离鸭子和它的飞行、呱呱叫等行为，一个好办法是提供飞行和呱呱叫的接口，然后在其他类中再实现这些接口，这样既保证了代码能够复用，又让不会飞和呱呱叫的鸭子没有继承到与他无关的相关方法

面向接口编程的核心就在于抽离一个类中在子类可能会变化的部分，并将他们设计成接口，这样一来能够起到解耦的作用
